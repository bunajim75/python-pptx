# Slide-layout research fixture corpus

This directory is the frozen corpus for package-level slide-layout research. It
starts deliberately empty; `MANIFEST.sha256` records that state with the exact
`# EMPTY: no fixture artifacts are committed yet.` marker. The validator tests
both that marker and the rest of the scaffolding, so an accidentally empty or
missing manifest is not accepted. Fixtures are added only by a later, focused,
reviewed change.

Fixtures belong directly (with no deeper subdirectories) under one of these
provenance buckets:

- `powerpoint-authored/`: created in desktop Microsoft PowerPoint;
- `powerpoint-normalized/`: opened and saved by desktop Microsoft PowerPoint
  after originating elsewhere;
- `synthetic-invalid/`: deliberately hand-built invalid packages used to prove
  failure behavior.

Scenario names use lowercase kebab-case, for example
`powerpoint-authored/two-content-layout.pptx`. Fixtures are frozen and must not
be generated by the production code under test. In particular, a file produced
by `python-pptx` cannot be relabeled as PowerPoint ground truth. PowerPoint-
authored and PowerPoint-normalized provenance must record the actual PowerPoint
version and platform honestly in the desktop-verification record.

Every `<scenario>.pptx` has an adjacent `<scenario>.json` provenance sidecar
and a `<scenario>.desktop.json` desktop-verification record. Every corpus
artifact is pinned in `MANIFEST.sha256`; the README and manifest itself are the
only control files excluded from the manifest. The validator rejects missing
or orphan sidecars, missing or orphan desktop records, unlisted files, and
manifest entries whose files are absent.

## Provenance sidecar

`<scenario>.json` is a JSON object with these fields:

- `schema_version`: `1`;
- `role`: the containing provenance bucket name;
- `date`: an ISO `YYYY-MM-DD` date;
- `description`: a non-empty description of the scenario;
- `source`: a non-empty account of how the fixture was obtained or built;
- `application`: the application that actually wrote the fixture bytes; it
  must identify Microsoft PowerPoint for the two PowerPoint roles and may be a
  non-empty string or `null` for a synthetic-invalid fixture.

## Desktop verification

`<scenario>.desktop.json` is a JSON object with `schema_version: 1`, `status`,
`date`, `application`, `version`, `platform`, and `notes`. Status is `passed`,
`failed`, or `not-run`. A completed check records an ISO date and the actual
desktop Microsoft PowerPoint application, version, and platform used. A
`not-run` record uses `null` for those four fields. `notes` is always a string.
Automated package tests do not substitute for this desktop verification.

## Manifest

Each non-control file appears exactly once as:

```text
<lowercase-sha256>  <path-relative-to-this-directory>
```

Generate candidate manifest lines from this directory with:

```sh
find powerpoint-authored powerpoint-normalized synthetic-invalid -type f \
  \( -name '*.pptx' -o -name '*.json' \) -print0 2>/dev/null \
  | sort -z | xargs -0 sha256sum
```

Review the fixture, both sidecars, and every manifest-line change together.
Hash drift is a failure and is never repaired automatically. The validator
also rejects malformed or duplicate manifest entries, missing or unlisted
files, orphan or missing sidecars, unsupported artifacts, invalid names or
directory placement, and invalid metadata. Fixture changes therefore require
intentional human review; generated PPTX files must remain temporary unless a
later fixture-focused change establishes honest provenance and freezes them.
